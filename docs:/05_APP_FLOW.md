# 镜市 MirrorMart — 应用流程文档 (App Flow)

> 版本 1.0 · 2026-03 · 状态: Phase 0 准备中

---

## 1. 全局流程总览

```
用户输入场景描述
        │
        ▼
   ┌─────────┐
   │  设定阶段 │ ← 解析输入 → 生成Agent → 构建平台环境 → 注入初始数据
   └────┬────┘
        │
        ▼
   ┌─────────┐
   │  模拟阶段 │ ← 蒙特卡洛循环: 对每个分支 { 初始化 → 逐步执行 → 记录 }
   └────┬────┘
        │
        ▼
   ┌─────────┐
   │  聚合阶段 │ ← 多分支结果合并 → 概率分布计算 → 关键事件提取
   └────┬────┘
        │
        ▼
   ┌─────────┐
   │  观察阶段 │ ← 用户查看概率分布 / 群体动态 / 个体旅程
   └────┬────┘
        │
        ▼ (可选循环)
   ┌─────────┐
   │  干预阶段 │ ← 修改条件 → [联网校准] → 重新模拟 → 对比结果
   └─────────┘
```

## 2. 设定阶段详细流程

```
用户输入
   │
   ├─ 自然语言描述 ──→ LLM 解析为结构化配置
   │                     ├─ 产品信息 (名称/价格/特点)
   │                     ├─ 目标场景 (电商/舆情/传播)
   │                     ├─ 目标平台 (小红书/淘宝/...)
   │                     └─ 预期问题 (想知道什么)
   │
   └─ 结构化表单 ────→ 直接使用
   │
   ▼
生成Agent群体
   ├─ 根据场景类型选择合适的Agent画像模板组合
   ├─ 每个模板实例化指定数量的Agent
   ├─ 在模板范围内随机化具体属性 (年龄/收入/性格参数有±范围)
   ├─ 生成社交关系网络 (随机图 + 可选的社区结构)
   └─ 为每个Agent生成自然语言人设描述 (用于LLM prompt)
   │
   ▼
构建平台环境
   ├─ 根据场景配置初始化选定的平台
   ├─ 注入初始内容 (品牌帖子/竞品商品/历史评价)
   ├─ 设定平台参数 (推荐算法权重/热度衰减速率)
   └─ [可选] 联网搜索真实数据校准初始参数
   │
   ▼
确认并开始
   ├─ 向用户展示: Agent群体概览 + 平台初始状态 + 模拟参数
   ├─ 用户可调整后确认
   └─ 进入模拟阶段
```

## 3. 模拟阶段详细流程

### 3.1 蒙特卡洛外循环

```
for branch_id in range(num_branches):
    │
    ├─ 1. 初始化分支
    │     ├─ 复制基础状态 (Agent状态 + 平台状态)
    │     ├─ 设定独立随机种子 (seed = base_seed + branch_id)
    │     └─ 创建分支事件日志
    │
    ├─ 2. 执行时间步循环 (内循环)
    │     for step in range(num_steps):
    │         ├─ 确定本步活跃Agent列表 (基于使用习惯+休眠状态)
    │         ├─ 对每个活跃Agent执行认知循环:
    │         │     perceive() → decide() → act()
    │         │     [条件触发: reflect()]
    │         ├─ 更新平台环境状态 (新内容/互动数据/热度变化)
    │         ├─ 检查终止条件 (所有Agent决策稳定 / 达到最大步数)
    │         └─ 记录本步事件和指标
    │
    └─ 3. 分支完成
          ├─ 保存最终状态快照
          ├─ 计算分支级指标 (总销量/传播范围/关键事件)
          └─ 标记分支结果类别 (爆款/平淡/危机/...)
```

### 3.2 单时间步 Agent 循环

```
Agent 被激活
    │
    ▼
PERCEIVE (感知)
    ├─ 获取当前所在平台的信息流
    ├─ 获取通知 (有人评论了我的帖子/有人@了我)
    ├─ 注意力过滤: 根据人设偏好筛选关注的内容
    └─ 输出: 感知摘要文本
    │
    ▼
RETRIEVE (记忆检索)
    ├─ 用感知摘要作为query
    ├─ 从记忆流中检索相关记忆
    ├─ 加权: recency(时间衰减) × importance(重要性) × relevance(相关性)
    └─ 输出: top-K 相关记忆
    │
    ▼
DECIDE (决策)
    ├─ 构造prompt: 人设 + 感知 + 记忆 + 当前计划
    ├─ 判断决策复杂度 → 路由到对应LLM层级
    ├─ LLM推理 → 生成行动决策
    ├─ 解析输出为结构化行动指令
    └─ 输出: action = {type, platform, target, content, ...}
    │
    ▼
ACT (行动)
    ├─ 在目标平台执行行动
    ├─ 平台返回执行结果 + 环境变更
    ├─ 记录行为到Agent的action_log
    └─ 将体验写入记忆流 (带时间戳+重要性评分)
    │
    ▼
[REFLECT] (反思 — 条件触发)
    ├─ 检查: 自上次反思以来的记忆重要性累积是否超过阈值
    ├─ 如果是: 用LLM综合近期记忆，生成高层洞察
    ├─ 洞察写回记忆流 (类型标记为 "reflection")
    └─ 洞察影响后续决策和计划更新
    │
    ▼
[CROSS-PLATFORM] (跨平台行为 — 条件触发)
    ├─ 检查: 决策是否涉及跨平台行为 (如"去淘宝搜一下")
    ├─ 如果是: 切换Agent当前平台
    ├─ 在新平台上执行额外的perceive→decide→act
    └─ 这是消费者旅程的核心: 小红书种草 → 淘宝比价 → 购买决策
```

## 4. 聚合阶段详细流程

```
所有分支完成
    │
    ▼
结果分类
    ├─ 对每个分支计算核心指标
    │     ├─ 产品总销量
    │     ├─ 内容总曝光/互动
    │     ├─ 正面/负面评价比例
    │     ├─ 传播深度 (最长转发链)
    │     └─ 购买转化率
    │
    ├─ 使用聚类算法将分支归类为结果类别
    │     ├─ 如: "爆款" / "小火" / "平淡" / "口碑危机"
    │     └─ Phase 0 可手动定义阈值分类
    │
    └─ 输出: 每个类别的概率 = 该类别分支数 / 总分支数
    │
    ▼
概率分布计算
    ├─ 每个核心指标: 计算均值/中位数/标准差/分位数
    ├─ 生成概率分布图数据
    └─ 标注异常值和关键转折点
    │
    ▼
关键事件提取
    ├─ 跨分支对比: 哪些事件在多数分支中出现
    ├─ 转折点识别: 哪些事件导致分支走向不同方向
    ├─ 影响力排名: 哪些Agent的行为影响了最多其他Agent
    └─ 输出: 关键事件列表 + 影响力排名
    │
    ▼
报告生成
    ├─ 概率分布摘要: "在5次模拟中，30%走向爆款，55%平淡，15%危机"
    ├─ 关键洞察: "价格是决定性因素——在3次模拟中，价格比较后放弃购买"
    ├─ 典型旅程: 选取1-2个有代表性的Agent旅程详细展示
    └─ 建议: "降价到49元或增加KOL背书可能提升爆款概率"
```

## 5. 干预阶段详细流程

```
用户查看结果后决定干预
    │
    ▼
选择干预类型
    ├─ 修改产品属性 (改价格/改卖点/改包装)
    ├─ 修改营销策略 (加KOL/改投放平台/改内容策略)
    ├─ 注入外部事件 (竞品活动/差评事件/行业新闻)
    ├─ 修改Agent群体 (增加特定画像/改变群体比例)
    └─ 自定义 (自然语言描述变化)
    │
    ▼
[联网校准] (可选但推荐)
    ├─ 根据干预类型自动搜索相关真实数据
    │     ├─ 改价格 → 搜索同品类价格带分布
    │     ├─ 加KOL → 搜索同品类KOL合作数据
    │     ├─ 竞品活动 → 搜索竞品近期营销活动
    │     └─ 舆情事件 → 搜索类似历史事件传播数据
    ├─ 将搜索结果注入到Agent的认知参数中
    └─ 目的: 让模拟参数有真实数据锚点
    │
    ▼
重新运行蒙特卡洛
    ├─ 使用修改后的条件重新生成模拟
    ├─ 同样的Agent群体 + 新条件
    └─ 运行相同数量的分支
    │
    ▼
对比分析
    ├─ 条件A vs 条件B 的概率分布并排对比
    ├─ 指标变化: 哪些指标改善了/恶化了
    ├─ 行为变化: Agent群体的行为模式有何不同
    └─ 建议: 基于对比给出方向性建议
```

## 6. Phase 0 简化流程

Phase 0 只保留核心路径，去掉所有可选步骤:

```
1. 硬编码场景 (facemask_launch.yml)
2. 硬编码20个Agent (从5个画像模板各实例化若干)
3. 构建简化小红书 + 简化淘宝环境
4. 运行5个分支 × 20个时间步
5. 每步: 每个Agent做 perceive → decide → act
6. 结果输出到JSON文件
7. 在Jupyter中分析结果

不做:
- 不做联网校准
- 不做用户输入解析 (硬编码场景)
- 不做干预循环 (分析后手动修改YAML重跑)
- 不做前端
- 不做反思机制 (Phase 1加入)
- 不做跨平台切换的自动触发 (手动在场景中预设)
```

## 7. 错误处理流程

```
LLM调用失败
    ├─ 网络超时 → 重试(指数退避, max 3次) → 仍失败 → Agent本步跳过
    ├─ 返回格式错误 → 尝试修复JSON → 失败 → 使用人设默认行为
    ├─ API限流 → 等待 → 重试 → 降级到更轻量模型
    └─ 所有LLM不可用 → 暂停模拟，等待恢复

分支执行失败
    ├─ 单Agent崩溃 → 标记该Agent为"离线"，继续模拟
    ├─ 平台环境崩溃 → 保存当前状态，尝试恢复
    └─ 全分支崩溃 → 标记该分支为"失败"，不计入聚合

结果聚合异常
    ├─ 有效分支 < 3 → 警告"样本量不足，结果仅供参考"
    ├─ 所有分支结果相同 → 警告"随机性不足，考虑增加Agent数或调整参数"
    └─ 分布极端异常 → 标记，建议检查场景配置
```
